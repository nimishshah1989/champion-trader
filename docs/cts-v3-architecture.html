CTS v3 — Architecture Vision

Champion Trader System
as a Strategy Platform

A flexible, code-first portfolio testing and simulation platform. Deploy any trading logic as a Python strategy class. One data source. Automatic backtesting. Live tracking from day one. A standardised evaluation page for every strategy.

The Goal: Convert CTS into a platform where we can deploy strategies as Python logic, test them instantly against 1Y / 3Y / 5Y of historical data, run walk-forward validation, and then track them live from the moment they are "locked" — with a fully standardised evaluation page showing all risk, reward, equity curve, benchmark comparison, and a complete transactions ledger with capital gains and charges. The platform is the infrastructure; each strategy is a plug-in.

Lock Strategy is the key moment. Once locked, the strategy's parameters are frozen and a live run begins. From that date forward, every daily result is tracked in real-time exactly as if capital were deployed — even if the strategy is still paper trading. You always know how a strategy is doing from its live date, not just in hindsight.

Data Layer — Single Source

All data comes from Kite (Zerodha). No other data sources, ever. Derived metrics (EMAs, ATR, TRP, stage classification, etc.) are computed per strategy from raw OHLCV — not stored globally.

Historical

Kite Historical API

Daily OHLCV bars for ~1,300 NSE stocks + indices. Fetched once, stored in a local SQLite bar store. Incremental updates daily at 5:30 PM IST. Up to 10 years of history available.

Strategy Layer — The Python Interface

Every strategy is a Python class inheriting from AbstractStrategy. The same class runs in backtest mode and live mode. No separate backtesting code. No drift between tested logic and deployed logic.

# backend/strategies/base.py — the contract every strategy must implement class AbstractStrategy(ABC): # ── Identity ────────────────────────────────────────────────────── name: str # "v2-setup" — DB key, URL slug display_name: str # "Champion Trader v2 Setup" description: str version: str # "1.0.0" — bumped when logic changes # ── Core methods (called identically in backtest AND live) ──────── @abstractmethod def generate_signals( self, bars: dict[str, pd.DataFrame], # {symbol → OHLCV dataframe} date: date, ) -> list[Signal]: ... # entry candidates for this date @abstractmethod def should_exit( self, position: Position, bars: dict[str, pd.DataFrame], ) -> ExitDecision | None: ... # None = hold @abstractmethod def size_position( self, signal: Signal, account_value: float, open_count: int, ) -> PositionSize: ... # qty, entry_price, stop_price # ── Params (used by backtest engine and UI sliders) ─────────────── def get_params(self) -> dict: # default parameters def get_param_ranges(self) -> dict: # {param: (min, max, step)} for future optimization

Portfolio Lifecycle

A "portfolio" is one instance of a strategy with a specific capital, params, and start date. You can run multiple portfolios of the same strategy (different params) simultaneously.

01
Create
Pick a strategy. Set capital, RPT%, max positions. Name the portfolio. Click Create.
02
Backtest
Engine runs 1Y, 3Y, 5Y backtests automatically using bar store history. Results ready in seconds–minutes.
03
Walk-Forward
Splits history into IS (in-sample) and OOS (out-of-sample) windows. Tests strategy on data it never saw.
04
Lock
Params are frozen. Strategy enters paper-live mode. Every result from this date is tracked as if real.
05
Go Real
Optionally connect to a real Kite account. Same signals, same sizing — now with real order placement.

Walk-forward validation: history is split into rolling windows (e.g. 12 months IS → test 3 months OOS, slide forward). This prevents overfitting — a strategy must work on data it was never trained on. Monte Carlo simulation (later): randomise trade order and entry timing thousands of times to stress-test the edge.

Technical Architecture

Backend

FastAPI + Python

GET /strategies — list all registered strategies
POST /portfolios — create a new portfolio run
GET /portfolios/{id}/backtest — fetch backtest results
GET /portfolios/{id}/live — SSE stream of live P&L
GET /portfolios/{id}/transactions — full ledger

Frontend

Next.js + TypeScript

/strategies — strategy selector grid
/strategies/[slug] — per-strategy hub
/strategies/[slug]/[portfolio-id] — evaluation page
Charts: Recharts (equity curve, drawdown, heatmap)
Live: SSE connection auto-reconnects every 15s

Backtest Engine

Event-Driven Loop

Pure Python. No backtesting library dependency — we own the loop. Bar-by-bar iteration over the bar store. Calls strategy.generate_signals() and strategy.should_exit() identically to the live scheduler. Fills stored as PortfolioTrade rows. Equity snapshots stored daily.

Database

SQLite → Postgres-ready

New tables: portfolios, portfolio_trades, portfolio_equity, portfolio_params.
Existing tables kept for v1 data reference.
Every backtest result is versioned by a hash of the params — changing a param automatically queues a new backtest.

What Stays the Same

Trading Rules

The v2 Champion Trader methodology is unchanged. Stage 2 + contraction + 5-day break, 5×ATR chandelier stop, 0.35% RPT — all stay as the v2 strategy module. Nothing about the logic changes; it just moves into the new structure.

Data Pipeline

Kite ingest at 5:30 PM. Exit pass at 5:40 PM. Entry pass at 5:45 PM. Scan at 5:50 PM. The scheduler continues running these jobs — they just call into the new strategy engine instead of the old live_jobs.py functions.

Dashboard + Actions

The daily workflow pages (Dashboard, Pipeline, Trades, Actions) remain unchanged. The new strategy pages are additive — a layer on top for deeper analysis, not a replacement for the operational flow.
