1 · What we set out to build — the "Explained-Blip" engine

An engine that treats each instrument as having a repeating behavioural fingerprint, watches for moments it behaves abnormally (a "blip"), explains the blip with a named economic mechanism, and trades only the explainable ones — using the explanation itself as the risk control.

The three moves:  normal → abnormal → why
  1. Fingerprint — learn each stock's own "normal" from Kite OHLCV (typical pullback depth, volume-price signature, volatility rhythm, earnings reaction).
  2. Blip — detect a statistical deviation from that stock's own normal (not a generic indicator).
  3. Explain — attach a falsifiable mechanism (e.g. post-earnings drift, index-rebalance flow, quiet accumulation / information-leakage à la Bajo 2010). Discard any blip with no nameable cause.
  4. Trade the explained ones — size and exit set by the mechanism's confidence and its failure condition (mechanism invalidated → exit). Slogan: "the why is the risk control."

Worked example (illustrative)

┌─ EXPLAINED BLIP ──────────────────────────────┐
│ BHARATFORGE · LONG                            │
│ Why: abnormal delivery-volume, no news →      │
│      quiet accumulation (info-leakage edge)   │
│ Normal: ~₹120cr/day, 3–4% dips on fading vol  │
│ Today: +4.5% on 3× vol, delivery 2.4σ high    │
│ Plan (set BY the reason, not a guess):        │
│   • Hold ~2–3 weeks (drift horizon)           │
│   • EXIT IF a high-volume reversal prints      │
│     (= "the buyer is gone") → thesis dead     │
│   • Confidence 0.62                            │
└───────────────────────────────────────────────┘

The core thesis: durable patterns repeat because their cause repeats; price alone cannot tell a causal pattern from a lucky coincidence — so evolve explanations, not shapes. (The council's central critique is that this reasoning, while appealing, does not actually remove the overfitting — see §4.)

2 · The "evolve git" — what ASI-Evolve actually is

GAIR-NLP/ASI-Evolve is an AlphaEvolve-style autonomous research framework: an LLM-driven multi-agent loop that evolves code to maximise a score you define.

Researcher

Proposes the next candidate program, reading a FAISS cognition store of seeded knowledge + an experiment DB of prior trials.

Engineer

Runs the candidate, collects structured metrics.

Analyzer

Distils each outcome into reusable "lessons" fed back into the store.

Sampler

Picks which prior trials to learn from via UCB1 / greedy / random / MAP-Elites islands (diversity-preserving).

You supply: a problem description, a baseline program, an eval.sh that returns a score (the fitness function), and optional cognition seeds.

The decisive property. ASI-Evolve's demonstrated wins are in domains with stable, verifiable, cheap, low-noise objectives — neural-architecture search (+0.97), data curation (+18 on MMLU), RL algorithm design (+12.5), drug-target prediction (+6.94 AUROC). It is, fundamentally, a score-maximiser over code. Whatever number eval.sh returns, it will relentlessly find code that maximises it — including by exploiting noise. The fitness function is the entire ballgame.

Five independent expert lenses were convened to answer one question: "Is this the best way to use ASI-Evolve's capability for CTS?" Each was briefed identically and instructed to be adversarial, not agreeable.

Where all five converged

1. Don't aim a score-maximiser at trade selection. CTS already built a composite ML alpha-score to pick setups, ran it out-of-sample (IC 0.03–0.08), and concluded "the edge is in the EXIT, not in selection." This proposal is a more elaborate, more persuasive version of the same bet. The added "mechanism" is a human-pleasing narrative, not causal proof — and it actively disarms the Deflated-Sharpe / plateau discipline that protects you, because you'll wave through a curve-fit blip "because it has a reason."
2. Wrong tool for the sub-task. ASI-Evolve shines at 10³–10⁶ cheap, low-noise, verifiable evals. A walk-forward backtest budget is ~10²–10³ total, and the signal is buried in noise — exactly where an LLM evolver has no edge over a plain sweep / Bayesian optimisation and far more failure surface (reward-hacking, candidate-DB poisoning by lucky trials, prompt drift). The mechanism scoreboard is fundamentally a GROUP BY mechanism over your existing attribution tables — statistics, not program search.
3. The genuinely best use of ASI-Evolve = evolve the EXIT & the cost/execution model. Your proven source of returns, on a verifiable, stationary-ish, high-N objective (every bar of every open trade is a decision) where the tool cannot fool you. Let it search the exit/trail structural form (chandelier multiple, time-stop, partial-scaling, regime-conditional trail) and calibrate a realistic slippage/impact model — against the existing walk-forward harness, writing only params back, never touching the runtime path.

The steelman (what's genuinely right in the idea)

  • "Explain-or-discard" is a real regulariser — requiring a nameable mechanism is a guard against blind multiple-comparisons mining, if the mechanism set is small and pre-registered.
  • "The why is the risk control" is sound — an economically-meaningful exit beats an arbitrary ATR line.
  • Two mechanisms are real, durable, and fillable in NSE: post-earnings-announcement drift (PEAD) (behavioural under-reaction, 50 years of global persistence) and index-rebalance flow (advance-noticed, measurable point-in-time).

5 · The council-revised recommendation

Track A — point ASI-Evolve where it earns its keep primary

Evolve the exit / sizing policy of v2 (and the cost model), not trade selection. 
input.md   :  "Improve OOS Calmar of v2 by evolving the EXIT/trail/sizing
              policy (structural form + params), net of NSE costs."
baseline   :  STRATEGY_V2 exit logic in engine/runtime/
eval.sh    :  run_strategy_wf.py  →  Deflated Sharpe / OOS Calmar,
              net of tiered slippage + circuit-lock, survivorship-aware
cognition  :  STRATEGY_HANDOFF.md §4 (what's already tested) +
              RESEARCH_VOLUME_MOMENTUM.md  (so it won't redo Tracks 1–5)
write-back :  params only → a new versioned config (v2.x) → A/B → adopt
              ONLY on a plateau. Runtime path never touched by the LLM.

MAP-Elites is genuinely useful here — maintain an elite exit-program per (volatility regime × trend regime) cell, not "diverse mechanisms."

Track B — if the blip idea is pursued, do it lean & honest secondary

  1. Phase 0 — one script, no module. Add a bhavcopy delivery% ingester (needed anyway), write fingerprint() + detect_blip() as pure functions, and a research script that ranks historical blips through run_strategy_wf.py. One report. If blip-rank doesn't correlate with forward edge — stop. Days spent, not weeks.
  2. Hard liquidity / ASM floor first. Restrict to ≥ ₹5cr/day ADV (ideally ₹25cr+) and exclude anything ever on ASM/GSM or with a ≤5% circuit band — before any search, so the toxic small-cap tail can't be mined.
  3. Keep only PEAD + index-rebalance flow. Demote delivery% to a confirmation filter; quarantine "accumulation / leakage" as a standalone trigger.
  4. Deterministic, rule-based explainer. No LLM in the decision path — it would break the parity gate. The LLM may propose mechanism hypotheses offline; humans + statistics promote them.
  5. Point-in-time everything. News/announcements are a lookahead minefield — an evolutionary searcher will ruthlessly exploit any leakage. Use purged/embargoed walk-forward around all event dates; next-open execution only.

Track 0 — do this first regardless highest EV

Forward paper-trade v2 on live Kite (backlog #1). v2 has never traded a live tick. Wire execution by hand (deterministic, not LLM-evolved): tick rounding, position reconciliation, adjusted-vs-raw data, circuit checks. This is the only true out-of-sample test, and no new alpha should be built on an unvalidated base.

6 · Guardrails (non-negotiable if any searcher touches a backtest)

  • Lock a final holdout before search; judge the winner there once. Never let the loop optimise on it.
  • Deflated Sharpe + Harvey–Liu haircut using the true trial count (ASI-Evolve logs every trial — feed N in).
  • PBO via CSCV / purged-embargoed CV → demand PBO < ~10–20%; reject above 50%.
  • Pre-register a small mechanism / hypothesis set with explicit failure conditions; cap the count.
  • Optimise a tail statistic (CVaR / 5th-percentile), not the mean — noise-mining lifts the mean but hurts the tail.
  • Plateau, not a peak — score the ±10–20% neighbourhood, the standard CTS already holds.
